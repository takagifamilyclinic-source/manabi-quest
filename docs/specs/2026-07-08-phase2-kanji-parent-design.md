## 設計① 教科選択＋バトル統合

- ホームの「⚔️ バトルにでかける」→ **教科えらび**(➗さんすう / ✏️かんじ)を挟む。
- 漢字も**算数と同じバトル形式**を再利用: 10問1セット→クリアでモンスター捕獲。報酬(図鑑・連続日数・捕獲)は共通。新しい報酬系は作らない。
- 解答UI: 算数=テンキー、漢字=**4択ボタン**。バトル進行ロジック(`battle.js`)はそのまま流用。
  - `battle.js` の `answer(battle, value)` は数値比較。漢字は選択肢の値で正誤判定するため、**問題オブジェクトに正解値を持たせ、answer判定を「value === question.answer(文字列も可)」に一般化**する(既存の数値ケースは `Number()` 経由のまま互換維持)。

## 設計② 漢字データ設計(案A・段階配信)

- 出題オブジェクトは算数と共通形式に `subject` を足す:
  `{ subject, skillTag, text, choices, answer, explanation }`
  - 算数は `subject:"math"`・`choices` 無し(テンキー入力)。漢字は `subject:"kanji"`・`choices`(4択)あり。
- **skillTag = 学年×形式**: `kanji-read-g1`(1年読み)/`kanji-write-g2`(2年書き)/`kanji-mean-g1`/`kanji-radical-g1`/`kanji-stroke-g1` 等。苦手分析の単位。
- **元データ**(全642字): `data/kanji/g<N>.js` に
  `{ kanji, grade, yomi: [かな読み...], meaning }` を1字ずつ。
  - 読み(漢字→かな): 元データの yomi から正解、ダミーは同学年の他字の読みから距離の近いものを自動抽出。
  - 書き(かな→漢字): 同じ元データを裏返し、正解=kanji、ダミー=見た目/意味の近い同学年字。
  - **読み・書きは全642字を初回から**用意。
- **意味・部首・筆順**: 追加データ(意味文・部首・画数/1画目)が必要。**1・2年から手作り配信→3・4年へ順次拡張**。最終的に全642字へ。
  - 部首: `radical` フィールド追加、4択。筆順: `strokes`(総画数)＋「1画目はどこ?」等の簡易4択(まず総画数当てから)。
- ダミー選択肢生成ロジックは純粋関数として分離(`kanji-quiz.js`)しテスト可能に。

## 設計③ 苦手自動調整

- `attempts`(既存 `{profileId, skillTag, correct, date}`)から**分野タグごとの直近正答率**を算出(直近N件重視)。
- `generateSession` を拡張し、**低正答率タグを高確率で出す重み付き抽選**にする。算数・漢字ともskillTag単位で共通に作用。
- ゼロ実績タグは標準重み。**重みに下限**を設け、苦手ばかりに偏らないようにする(嫌気防止)。
- 純粋関数 `weightBySkill(skills, attempts, profileId) -> 重み配列` として分離しテスト可能に。

## 設計④ 親ページ(隠しボタン＋4桁PIN)

- 入口: プロフィール選択画面の隅に小さな⚙、**長押し(約1秒)**でPIN入力画面。初回はPIN設定(確認2回)。
- 中身(子ごと):
  1. 学習履歴(連続日数・セッション数・捕獲数)
  2. **苦手トップ5**(タグ別正答率の低い順・試行数つき)
  3. PIN変更
  4. **学習記録の書き出し**(state のJSONをテキストエリアに出しコピー可能に。将来のClaude連携素地)
  5. リセット(子別 or 全体。実行前に確認)
- PINは `state.settings.pin`(端末内)。注記「PINリセットは記録も消える可能性」を表示。
- 親ページUIは子ども向けと分け、落ち着いた配色・通常サイズ文字。

## 設計⑤ データ設計の変更

- `state` に `settings: { pin: null }` を追加。schema **version 3**。
  - v2以前のデータは `load()` で defaultState にリセット(第1弾同様、安全側)。**将来の本移行TODOは設計書TODO順守**。
- `attempts` は既存構造のまま苦手分析に使用(subjectは skillTag 接頭辞 `kanji-`/算数タグで判別)。
- 漢字パック/元データは `data/kanji/g<N>.js` で同梱。**sw.js の ASSETS に追加し CACHE を上げる**。

## ファイル構成(追加・変更)

```
data/kanji/g1.js … g4.js   # 学年別の漢字元データ(kanji,yomi,meaning[,radical,strokes])
js/kanji-quiz.js           # 漢字出題(読み/書き/意味/部首/筆順)＋ダミー選択肢生成(純粋)
js/weakness.js             # 苦手重み付け weightBySkill(純粋)
js/math-gen.js             # generateSession を教科横断＋苦手重み対応に拡張
js/battle.js               # answer の正誤判定を数値/文字列一般化
js/state.js                # settings.pin 追加・version 3
js/app.js                  # 教科えらび・漢字4択UI・親ページ画面・隠し入口
data の各テスト             # kanji-quiz / weakness / state / battle
sw.js / manifest.json      # 漢字アセット追加・CACHE更新
```

## エラー・例外への備え

- 漢字元データの読み欠落/重複はテストで検出(全字に yomi≥1・kanji一意)。
- ダミー選択肢が正解と重複しない/4択そろう保証をテスト。
- PIN未設定時の親ページ入口は「設定」フローへ。PIN照合失敗の回数制限は軽く(こども誤操作防止が目的)。
- 苦手重みの下限で全問苦手化を防止。

## テスト

- `kanji-quiz`: 出題形式ごとに正解が元データと一致・選択肢4つ・正解を含む・ダミー重複なしを乱数ループ検証。
- `weakness`: 低正答率タグの重みが高くなる/ゼロ実績は標準/下限が効くことを検証。
- `state`: settings.pin・version3・v2リセットを検証。
- `battle`: 文字列選択肢の正誤判定(既存の数値互換を壊さない)。
- ブラウザ: 教科えらび→漢字バトル→捕獲、親ページのPIN・苦手トップ5・書き出し。
